# Deep Dive

特定テーマの深掘り分析(設計書7.4)。日常的に実行せず、本人の明示的な依頼時のみ。

テーマ例: FDE型ビジネスはSIerに適用できるか / AI時代のエンジニア生産性評価 / Claude Codeを組織展開する方法 / Palantir型垂直統合の構造

## 手順

1. 引数のテーマを確認する。曖昧なら、何を意思決定したいのかを本人に1問だけ確認する
2. 材料を集める:
   - リポジトリ内: 過去レポート(該当テーマのみ)、decisions/、hypotheses/、self/current-focus.md
   - 外部: WebSearch / WebFetch で一次情報を優先して調査する(取得したURL・発行元・公開日を必ず記録)
3. 複数情報を比較し、対立する見解があれば両論を併記する
4. 事実と推測を明確に分ける(設計書8.2)。すべての主要な主張に出典を付ける
5. 保存前に source-validator のチェックリスト(形式チェック)を実行する
6. **独立レビュー(裏取り+校閲)を必ず実行する**(EV-20260713-01 で承認、evolution/changelog.md 参照)。執筆者とは別に Agent(general-purpose)を起動する。(a) 事実の裏取り: 主要主張ごとに出典を照合させ、「執筆者の主張を疑い、反証を試みる」姿勢で判定(確認済み / 要修正 / 出典不足)と根拠URLを返させる。疑わしい主張は WebFetch で原文を再確認する。(b) 校閲: 誤字脱字・見出し構成・config/output-preferences.yaml との整合(段落長・箇条書き・空行等)を確認させる。指摘を反映して修正する
7. `reports/deep-dive/YYYY-MM-DD-<テーマ短縮名>.md` に保存する

## レポート構成

1. 問い(何を意思決定するための分析か)
2. 確認された事実(出典付き)
3. 論点の整理(複数情報の比較。対立点を明示)
4. 自分の業務・組織・事業への適用分析
5. 仮説(hypotheses/ への登録候補があれば明示)
6. 意思決定案(選択肢・前提・リスク・見直し時期)— decision-capture への接続を促す
7. # Review(独立レビューで確認した点・修正した点・残る限界)
8. # Sources / # Feedback

記事IDは `DD-YYYYMMDD-NN`。
