## DB-20260714-01 Google Cloud「Cloud Run Sandboxes」:生成AIコードの隔離実行環境がパブリックプレビュー

**要点: 「AIが書いたコードを信用せずに実行する」ためのマネージドな仕組みが、主要クラウドの標準機能になりつつある。**

### 確認された事実

Google Cloudが、生成AIによる信頼できないコードを隔離して実行する「Cloud Run Sandboxes」をパブリックプレビューとして公開した。

- **認証情報と環境の隔離**: 環境変数へのアクセスとメタデータサーバ呼び出しを制限する。
- **ネットワーク制御**: アウトバウンド接続はデフォルト拒否。明示的な許可時のみ通信できる。
- **ファイルシステム保護**: 読み取り専用で動作し、変更はメモリ上のみ・実行終了時に破棄される。

次バージョンの Agent Development Kit および ComputeSDK で対応予定。料金・リソース上限は記事に記載がない。

### なぜ重要か

**AIエージェントの組織展開で最大の統制課題である「AI生成コードの実行リスク」に、クラウド側が標準解を用意し始めた。**

- 自前でコンテナ隔離・ネットワーク統制を作り込まなくても、マネージドで調達できる選択肢が増えた。
- Anthropic・OpenAIのエージェント実行環境や E2B 等の専業と同じ領域であり、「エージェントのサンドボックス」は差別化機能から標準機能へ移行しつつある。

### 自分の業務への示唆

**Claude Code等を組織展開する際の「実行環境の統制」を、自社標準にどう書くかの材料になる。**

- AI開発標準に「AI生成コードは隔離環境で実行する」という原則を置き、その実装手段としてこの種のマネージドサンドボックスを位置づけられる。
- 顧客案件でエージェントを組み込む際、セキュリティ説明(隔離・ネットワーク・データ持ち出し)の語彙としてそのまま使える。

### 推奨アクション

**AI開発標準の「実行環境」節の要件を1枚に整理する。**

- 隔離・ネットワーク・認証情報・ファイルシステムの4観点で、自組織のエージェント実行要件を書き出し、Cloud Run Sandboxes / 既存手段と突き合わせる。
- 工数: 1時間。
- 完了条件: AI開発標準ドラフトに「実行環境の統制」の節が1つ増えている。

## DB-20260714-02 DoorDash「Ask DoorDash」:LLM単体に頼らないAIアシスタントの設計と成果

**要点: 「LLMに全部やらせない」構造化されたエージェント設計が、チェックアウト転換率+24%という事業数値で裏付けられた。**

### 確認された事実

DoorDashがAI会話型ショッピングアシスタント「Ask DoorDash」のアーキテクチャを公開した。

- 構成は、オーケストレーション層(Assistant Runtime)+特化エージェント群+MCPベースのツール層+3種のメモリ(長期・セッション・エージェント)+自動評価基盤。
- ビジネスロジックをプロンプトに埋め込まず、カタログ検索・カート・チェックアウト等を共有MCP層のツールとして公開する。LLM呼び出し不要な処理は決定的なアクションで処理する。
- 公表された成果(自社発表の数値): コンシューマーメモリ機能により、食料品チェックアウト転換率 約+24% / バスケットサイズ +17% / 会話ターン数 -7%。オープンエンドな質問に限り、レストラン発見の転換率 +15%。
- 評価は1日2,000件以上の自動評価を実施し、リグレッション検査は6時間から20分に短縮された。

### なぜ重要か

**「エージェント=プロンプトの工夫」ではなく「エージェント=システム設計」であることを、事業数値付きで示した事例。**

- ロジックのツール化(MCP)、決定的処理の分離、メモリ設計、自動評価という構成は、業種を問わず流用できる設計パターン。
- 特に自動評価基盤をアーキテクチャの一部として最初から組み込んでいる点が、PoC止まりの社内AIとの違い。

### 自分の業務への示唆

**顧客提案・社内展開の両方で「AIアシスタント案件の標準アーキテクチャ」の参照例になる。**

- SIerとしてAI導入案件を受ける際、「LLM+プロンプト」ではなくこの5要素(オーケストレーション/エージェント/ツール層/メモリ/評価)で見積り・体制を組む根拠に使える。
- 数値は自社発表(二次情報経由)なので、提案書に使う場合は「DoorDash社公表値」と明示する。

### MBA視点での示唆

**AI投資のROIを会話品質ではなく転換率・バスケットサイズで測っている点が本質。**

- Unit Economics に直結するKPIへAI施策を接続しており、「AIの成果をどの経営指標で測るか」を先に決める設計の好例。

### 推奨アクション

**この5要素をチェックリスト化し、次のAI案件・社内AI企画の設計レビューに使う。**

- 工数: 30分。

## DB-20260714-03 「Minimum Viable Governance」:開発者に嫌われないコンプライアンス統制の実装例

**要点: 統制は「全部盛りのルール」ではなく「重要な5項目+自動フィードバック」から始めるべき、という実践論。**

### 確認された事実

InfoQが、Sevdesk(独・会計ソフト企業)のエンジニアリングマネージャー Davide de Paolis 氏の講演を公開した。

- クラウド統制のタグ付けを25〜30項目ではなく、5つ(所有者/サービス/ユーザーデータ/重要度/機密性)に絞る「minimum viable governance」を提唱。
- 違反検知は AWS Config → EventBridge → Lambda → Slack通知(Amazon Q chatbot経由)で自動化し、Slackで当該チームへフィードバックする。
- 主張は「プラットフォームチームは開発者を指示するのではなく支援する」「コンプライアンスは一夜では達成されない」。

### なぜ重要か

**AI利用ガバナンスにもそのまま適用できる「統制の最小設計+自動フィードバック」のパターン。**

- 統制側とエンジニアの関係が悪化する典型要因(ルール過多・手動指摘)への具体的な対処例。
- 「まず本当に重要な項目を絞る→検知と通知を自動化→対話で改善」の順序は、Claude Code利用統制の設計順序としても使える。

### 自分の業務への示唆

**Claude Code組織展開の統制設計を「最小5項目」から始める、という設計指針が得られる。**

- 例: 利用者(所有者)/対象リポジトリ/扱うデータ区分/レビュー必須度/ログ保全、のような最小タグ設計から着手する。
- 違反や逸脱は人が指摘するのではなく、自動検知してSlack等で本人に返す仕組みを標準に含める。

### 推奨アクション

**AI利用統制の「最小タグ5項目」案を書き出す。**

- 工数: 30分。
- 完了条件: AI開発標準ドラフトに、統制タグ5項目とその自動検知方法の案が書かれている。

## DB-20260714-05 マルチリージョンAWS APIから「隠れた1往復」を除去した実装記録

**要点: 何年も前の技術的制約への回避策が、いつの間にか可用性のボトルネックになっていた——「回避策の定期再点検」の教訓。**

### 確認された事実

InfoQが、マルチリージョンAWS APIの改善事例を公開した。

- 従来はSigV4署名がリージョンに固定されるため、実リクエスト前にリージョンを知るための「pre-flight discovery call」が全クライアントに組み込まれており、p90で75ms(同一リージョン)〜最悪1秒のオーバーヘッドとリージョン障害時のフェイルオーバー阻害要因になっていた。
- SigV4a(ECDSA-P256の非対称署名)への移行でリージョン集合に対して署名できるようになり、事前呼び出しを除去した。
- 署名コードの変更自体は小さいが、ファイアウォール許可リストや旧SDK対応などの調整に約6か月(構築3か月+並行稼働3か月)を要した。
- 著者の教訓は「ロールアウトの問題はアルゴリズムではなくプロセスに起因した」「クラウドの回避策は現在の提供機能と定期的に突き合わせよ」。

### なぜ重要か

**「過去に正しかった回避策」が負債化する構造と、その除去コストの実測値(6か月)が示されている。**

- 技術判断そのものより、レガシー除去の大半が調整コストであるという点が、見積り・計画の参考になる。

### 自分の業務への示唆

**保守案件・内製支援で「回避策の棚卸し」を定期作業として提案できる。**

- 顧客システムの「昔の制約による回避策」を洗い出し、現在のクラウド機能で置換できるかを点検するメニューは、継続収益型の役務になりうる。

### 推奨アクション

なし(継続監視)。教訓のみストックする。
